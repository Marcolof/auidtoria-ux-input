Evaluación UX y criterios funcionales para domicilio

Documento navegable para comparar propuestas de captura de domicilio, validar inputs y fundamentar la recomendación UX sin cambiar el alcance del MVP.

Requerimiento base: Calle · Altura · Piso · Departamento Foco: claridad del dato Discusión: layout vs. estructura

1. Definición del problema

El requerimiento actual define cuatro inputs independientes para domicilio: Calle, Altura, Piso y Departamento. La discusión no debería centrarse solamente en si los campos se muestran en una misma fila o en filas separadas, sino en si cada campo captura un dato claro, no duplicado y validable.

1

Problema de estructuraSi el campo “Piso” permite ejemplos como 1B, puede absorber información que corresponde al campo “Departamento”.

2

Problema de consistenciaEl cliente busca mantener un patrón visual full-width porque el resto de la web no usa campos agrupados en una misma fila.

3

Objetivo UXConservar una captura estructurada del domicilio, reduciendo ambigüedad, errores de carga y validaciones confusas.

4

Alcance MVPNo se agregan campos nuevos como torre, lote, manzana o unidad funcional. Solo se definen criterios para los campos existentes.

Clave para la conversación: agrupar “Calle + Altura” en una fila es una decisión de layout. Mantener “Calle” y “Altura” como inputs separados es una decisión de estructura de datos.

4. Casuísticas y principios UX

Separación semántica

Si el requerimiento define Calle, Altura, Piso y Departamento como inputs independientes, cada campo debe tener una responsabilidad clara.

Progressive disclosure

El switch es válido si se quiere ocultar información no relevante para usuarios que no necesitan cargar piso/departamento.

Consistencia contextual

La consistencia visual importa, pero no debería forzar que datos distintos se mezclen en un mismo campo.

Prevención de errores

La interfaz debe evitar ejemplos ambiguos que habiliten duplicación: Piso “1B” + Departamento “B”.

MVP responsable

No hace falta agregar torre, lote o manzana. Sí hace falta definir validaciones mínimas para los campos existentes.

Claridad de microcopy

Los ejemplos deben enseñar dónde va cada dato: Piso = PB, 1, 2. Departamento = A, B, 1B.

Casuística crítica: “1B” puede ser un departamento completo en algunos edificios, pero si existe un campo separado llamado “Departamento”, conviene reservar ese ejemplo para Departamento, no para Piso.

6. Conclusión / recomendación

Recomendación UX fuerte

Usar la Propuesta 2: Calle + Altura agrupados, switch “Agregar piso y departamento” y Piso + Departamento separados. Es la más clara y eficiente para el usuario.

Recomendación de compromiso

Si el cliente prioriza consistencia visual full-width, usar la Propuesta 3: todos los campos apilados, con Piso como selector de lista cerrada (PB y 1 a 20) para evitar la ambigüedad sin texto de ayuda.

No recomendado

No mantener el supporting original de Piso con el ejemplo “1B” si Departamento sigue existiendo como campo independiente.

Frase para presentar al cliente: “Podemos mantener la consistencia visual del resto de la web, pero conviene conservar la separación funcional entre Calle, Altura, Piso y Departamento. Lo importante es no sugerir que un campo puede absorber información de otro, porque eso genera ambigüedad y menor calidad del dato.”
